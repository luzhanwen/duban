# UI 设计标准

> 最后更新：2026-07-22

这份文档用于保护「读伴」现有的界面气质和组件边界。后续开发新功能、修 bug 或接入新格式时，都应先检查这里的标准，避免因为功能代码改动破坏整体视觉、比例和交互稳定性。

## 设计气质

- 视觉定位升级为「现代数字书斋」：米纸质感、朱砂猫章、现代藏书票、淡墨文字和安静留白共同构成主视觉。
- 核心氛围是安静、陪伴、私藏感，而不是冷感 SaaS、厚重仿古卷轴、儿童绘本化或电商书架感。
- 传统元素只作为质感和记忆点出现，不能牺牲现代软件的清晰层级、点击效率和布局稳定性。
- 猫形象是安静的伴读者，承担整理、陪读、空状态和完成反馈，不做夸张表情包或强拟人动作。
- 页面首屏应直接进入可用的书架、阅读、笔记或设置体验，不做重复的品牌 banner。

## 色彩标准

- 主背景使用米纸底色 `#F7F1E6`，主要表面使用 `#FBF6EC`，次级分区和悬停底色使用 `#EFE5D6`。
- 全局品牌强调色使用朱砂 `#B94132`，只用于 Logo、当前状态、关键操作、进度、印章和少量图标。
- 朱砂红建议控制在 3%–7% 的视觉面积内，不能大面积铺满，也不能抢占正文阅读内容。
- 文字使用淡墨体系：主墨 `#2B2622`、次墨 `#5F554A`、弱提示 `#8A7664`。
- 边框使用柔和纸边：普通边框 `#E3D6C6`，选中或重点边框 `#D3C1AD`。
- 次级工具操作使用米纸底、淡米边框和墨色文字，例如上传、导入样例、配置入口，不要和阅读主按钮抢视觉权重。
- 辅助状态色可以使用暖杏、低饱和橄榄、浅朱砂，但必须低饱和、低面积。
- 禁止用木质托盘、厚重渐变、3D 书脊、高光反光等拟物元素来制造“书架感”。

## 背景与纸纹标准

- 所有主要页面应有轻微纸纹背景。纸纹只能作为氛围层，不承担信息表达。
- 纸纹透明度建议控制在 3%–6%，不能影响正文、按钮、进度条或书名识别。
- 小字号文字下方不要叠加强纹理；卡片、弹窗、输入框可使用更浅的纸面色与全局背景拉开层次。
- 纸纹不能有明显方向性、水印、暗角或强颗粒。

## 字体标准

- 产品界面统一通过 `--font-app-cn` 使用同一中文字体家族；当前无授权依赖的评审基线为 macOS `Songti SC`，并依次回退到 `STSong`、思源宋体/Noto Serif CJK 和系统 serif。
- 页面标题、正文、按钮、输入框、标签和阅读辅助区默认继承 `--font-app-cn`，不要在业务组件里另写中文字体栈。代码、模型标识、API Key 和技术诊断继续使用等宽字体。
- 品牌中文「读伴」固定使用项目内置的 `Duban Brand Script` 两字子集，保留国风笔意、明确起收笔和适度墨色重量；不得回退为纤细随手写风格。
- `Duban Brand Script` 只用于品牌名称“读伴”，不能用于标题、按钮、提示或正文；界面正文和按钮必须以稳定、可读的字体为主。
- 书名和重要标题优先使用克制的宋体/serif 字体栈，例如 `--font-literary-ui` 或 `--font-display-cn`。
- 正文说明、按钮、数据标签需要清晰易扫，不要全部使用强烈书法字体。
- 英文 `DUBAN` 使用清晰的拉丁字体作为辅助识别，字距保持 `0`，不抢中文字标权重；默认只在竖版出现，横版需要时显式开启。
- 字号层级要稳定：页面标题 > 书名 > 作者/摘要 > 标签/辅助信息。
- 不使用 `vw` 直接缩放字体；长标题用行数限制、截断或自然换行处理。
- 付费字体候选优先评估汉仪君黑（全局产品字体）与汉仪玄宋（更偏阅读、书卷气方向）。只有在授权书明确覆盖桌面软件/App 嵌入、Webfont、安装包与更新包分发、字体文件随包交付和必要子集化后，才能把字体文件加入仓库或正式构建。

## 品牌 Logo 规格

- `horizontal` 横版：图形在左、中文字标在右，默认不显示 `DUBAN`，用于顶部导航、标题栏和横向品牌签名；需要英文辅助标识时通过 `showLatin` 显式开启。
- `vertical` 竖版：图形在上、中文字标与 `DUBAN` 在下，用于开屏、启动页和需要居中呈现品牌的场景。
- `compact` 简版：只显示图形，并保留无障碍名称“读伴”，用于首次设置共享 Logo、小尺寸入口、App 图标语境和空间受限区域。
- 三种规格必须由 `BrandLogo` 的 `variant` 输出，共享同一个 `LogoMark`，不得在业务组件里重新拼字标、英文名或尺寸。
- Logo 默认透明背景；除系统 App 图标或明确的实体载体外，不额外套米白方块、圆角底板、边框或阴影。
- 横版、竖版和简版可按场景缩放，但图形不得拉伸，中文与英文的相对层级必须保持稳定。

## 布局标准

- 书架当前采用封面优先网格，书本封面是第一视觉主体，页面标题和辅助信息必须轻。
- 网页版导航与书架使用全视口框架，只保留 24–64px 响应式安全边距；不得直接复用桌面 App 的固定最大内容宽度，在 2K/4K 浏览器中制造大块对称空白。
- 全视口不等于拉伸书卡：桌面书卡维持稳定宽度并由网格按空间增加列数，书少时自然从起始侧排列。
- 封面网格的列宽、间距和页面左右内边距要稳定，避免宽屏里书本漂浮或挤成信息表。
- 页面顶部只保留必要导航、标题元信息和上传/测试导入等工具操作，不再增加大面积 banner。
- 导读、正文、读后和完成页属于固定视口工作场景：页面骨架本身不得出现纵向滚动条，标题、状态和主操作保持在视口内；只有导读正文、书页、对话时间线、目录等会持续增长的内容区域允许独立滚动，并使用 `overscroll-behavior: contain` 防止滚动穿透。
- 绝对定位只能用于菜单、overlay、轻量提示和动效层，不能遮挡封面主操作或造成误触。
- 移动端优先使用自然高度，不强行套桌面固定高度。

## 藏书票式书卡标准

- 书架页的核心组件应逐步从单纯封面卡升级为现代藏书票：纸卡、细边框、书封、登记信息、阅读状态、朱砂进度线和右下小藏书章。
- 标准桌面书卡宽度建议 180–200px，高度建议 280–320px；移动端优先双列，宽度 156–172px。
- 卡片底色使用 `#FBF6EC`，边框使用 `#E3D6C6`，圆角控制在 12–16px，阴影轻到像纸片放在桌面上。
- 进度条使用 2–4px 高，轨道 `#E6DAC9`，进度 `#B94132`。
- 阅读状态文案优先使用「未启卷」「阅读中」「已读 2%」「读至第 42 页」「上次翻阅：昨日 21:16」「读伴藏书 · 第001卷」这类纸本登记语气；右下状态章区分「今日未读 / 今日在读 / 今日完成 / 全书读完」，避免把“今天打开过书”误写成“今天已经读完”。
- `readingDays / lastReadAt` 只证明发生过阅读活动；「今日完成」必须来自当天的阅读项完成时间，不能仅凭打开书籍、保存页码或产生阅读日判断。
- 避免 `0%`、`Processing`、`Untitled`、`No Data` 等工程化或空洞状态描述。
- PDF 优先使用第一页生成封面缩略图；MOBI 或封面生成失败时使用文字封面兜底。
- AI 封面生成先搁置；后续若接入，应作为“设定读伴”完成后的可选产物：优先服务无真实封面的书，默认不覆盖 PDF 原封面；生成风格保持米纸、淡墨、朱砂和藏书票气质。
- 封面本身是主操作入口：未确认进入书籍信息，未设定读伴进入「设定读伴」，已生成阅读计划进入阅读。
- 次级操作收敛到封面下方的小省略号菜单，不压在封面右上角，也不抢主操作。
- 封面比例要稳定，长书名、英文书名、PDF 扫描封面和 MOBI 文字封面都不能撑破网格。

## 基础组件标准

- 顶部导航高度保持 64–76px，背景使用 `#FCF8EF` 或半透明米纸色，分割线使用柔和米色边框。
- 顶部导航当前态使用浅米底和朱砂短下划线，导航文案优先使用「藏书」「设置」等更贴合书房语境的词。
- 工具按钮图标必须与动作一一对应：茶杯表示暂歇/会客，圆形勾选表示完成，对话气泡表示读伴介入，打开的书表示藏书或返回书架，印章只用于藏书登记。
- 同一个语义可以复用同一图标，不同语义不得因为“看起来有国风感”而共用印章；未知图标名不得静默回退成其他有明确含义的图标。
- Primary 按钮使用朱砂红底、米白文字、轻微朱砂阴影。
- Secondary 按钮使用米纸底、柔和边框、淡墨文字。
- Ghost 操作用无底或极浅底加朱砂文字。
- Danger 操作使用朱砂描边或深朱砂文字，不使用高饱和警报红大面积铺底。
- 输入框高度保持 36–42px，背景 `#FCF8EF`，边框 `#E3D6C6`，focus 使用朱砂描边，不使用系统蓝色。
- 空状态优先使用猫形象或淡朱砂小章、一句短标题、一句说明和一个主按钮。推荐「案头尚空，先加入第一本书」这类表达。

## 交互与动效标准

- 书架封面可以有轻微入场、hover 抬升、缩放和鼠标跟随旋转，但表面应偏哑光纸纹与轻阴影；避免扫光、亮斑、发光边缘和玻璃卡片质感。
- hover/focus overlay 只用于补充书名、作者、格式或操作暗示，不能改变网格尺寸。
- 鼠标跟随旋转只在非触控指针和非 reduced-motion 环境启用。
- 菜单打开时不能触发封面主操作，也不能让封面 hover 状态变得不可控。
- 页面级过渡可以用于读前导入到正文阅读等明确场景，但应优先使用 2D 纸面、位移、透明度和轻阴影；避免大幅 3D 翻转、滤镜和多层复杂关键帧。
- 页面级过渡不能阻塞内容可用性：底层目标页面应尽早挂载，活动记录、滚动定位等副作用要避开动画中段。
- 必须支持 `prefers-reduced-motion`，动效不能成为使用前提。
- 触控设备没有可靠 hover，关键操作必须通过封面点击和菜单可达，不要只藏在 hover overlay 里。
- 成功状态可以出现一次轻微“盖章”感反馈；整理中可用轻微翻页或进度推进，不使用强烈旋转 loading。
- 陪读新消息使用聊天方向一致的轻量入场：用户从右、读伴从左、过程记录从下方淡入；历史消息初次载入不逐条重播，入场不能改变卡片尺寸或滚动定位。

## AI 内容展示标准

- 模型回答必须通过受控 Markdown 解析器展示，至少正确处理段落、列表、粗体、斜体、引用和行内代码，不能把 `**`、`*` 等语法标记直接暴露给用户。
- AI Markdown 不执行原始 HTML，不使用 `dangerouslySetInnerHTML`；链接沿用解析器的安全 URL 规则，并在新窗口打开。
- Markdown 的段落、列表和标题应保持对话气泡内的紧凑字号与间距；粗体只能做行内强调，不能继承卡片标题的块级样式。
- 长 Markdown 内容必须兼容时间线折叠与展开，不能因为列表节点替换了单一段落就失去收起能力。

## 读伴界面标准

- 读伴是导读、正文、问答、笔记和读后记录之上的连续阅读层，不是独立聊天应用或侧栏后台。
- 同一视口只保留一个主要身份信号。完整或标准形象已经出现时，其他回答和记录优先使用排版方向、细线或简化印记，不重复头像和“读伴”标题。
- 导读、读中侧栏和读后各自最多保留一个主要承载面。内部层级优先使用留白、分隔线、字重和位置，不做卡片套卡片。
- 用户问题靠右、读伴回答靠左；笔记、导读线索、回想和完成记录居中或沿记录线呈现，不把所有类型做成同一种气泡卡。
- 阅读页优先保证正文和当前问题/回答。历史记录、usage、来源和低频动作逐级收起，不能让时间线像诊断日志。
- 读伴形象必须提供完整、标准和简化印记三种专门规格，并使用透明背景；小规格不能只把完整图等比缩小。
- 新形象应与米纸、淡墨、少量朱砂和猫章品牌属于同一体系，但不能直接复制 App Logo，也不能依赖独立彩色底板、拟物光晕或儿童贴纸感成立。
- 设置继续作为阅读现场内的轻量弹层。四项硬规则、软记忆、具体说明和保存反馈保持清楚，不改成独立设置页或抽象人格选择器。
- 「整理这本书」承担跨章节成果的下游管理：记忆列表按阅读顺序显示来源，编辑区与列表分栏，窄屏改为上下排列；不在该页重新制造聊天入口或自动生成新记忆。
- 场景动效只交接同一读伴、当前问题或保留记录，不通过弹跳、呼吸、红点和主动冒泡吸引注意。正文静默状态必须完全静止；准备/回答只在真实任务运行期间显示轻量反馈，记录/完成只播放一次，错误/离线使用静态降级。
- 视觉状态必须由 `CompanionShell` 的真实 loading、保存、完成、错误和联网信号驱动，并复用完整、标准、印记三种现有规格；不得随机换表情、另建状态头像或用定时器制造状态。
- 所有状态动画必须服从 `prefers-reduced-motion`；关闭动画后仍要保留取消操作、错误文案、离线说明和键盘可用性。
- 窄窗口中读伴使用覆盖层、抽屉或单列展开，不参与压缩正文；收起和重新打开不能丢失草稿、滚动、引用和生成状态。
- 阅读页收起读伴后只保留透明背景的简化猫耳书页印记作为唤醒入口，不在工具栏重复显示「打开读伴」文字按钮；印记至少提供 44px 点击区域、键盘焦点和明确的无障碍名称。
- P7.8 的审计和定稿过程保留在 [COMPANION_UI_AUDIT.md](./COMPANION_UI_AUDIT.md) 作为历史；后续界面以本文件的现行规则和 `COMPANION_ACTIVE_READING_PLAN.md` 的功能边界为准。

## 功能接入标准

- 新功能优先进入已有信息架构：顶部工具区、卡片菜单、设置页或阅读器侧栏。
- PDF 阅读器保留“滚动”和“翻页”两种明确模式：滚动模式按宽度排布连续页面；翻页模式必须同时适配可用宽度和高度，完整单页居中显示且阅读面板不出现纵向滚动。
- 滚动与翻页模式都允许收起读伴侧栏进入专注阅读；收起侧栏不能改变当前页码、PDF 比例、文本层或高亮坐标，重新打开侧栏后页面应按新可用区域自适应。
- 桌面窄窗口不能把标题压成逐字竖排，也不能让侧栏参与压缩 PDF：900–1180px 下标题与工具栏分为两行，默认进入专注阅读；重新打开读伴时使用覆盖式侧栏。
- PDF 滚动模式必须维持可读宽度下限；窗口低于该宽度时允许书页内部横向查看，不能继续等比缩小到正文不可辨认。
- 阅读计划内以“本节页码”为主：当前阅读项的 `startPage` 显示为本节第 1 页，仅在本节第一页旁轻量提示一次原书来源页码；顶部进度和读伴状态不重复强调。笔记、高亮、AI 上下文和恢复进度仍存原始页码。
- 翻页入口统一更新同一份页码状态：顶部前后页、键盘方向键、书页边缘点击、触摸左右滑动和触控板横向手势不得产生不同进度结果；连续手势需要阈值与防连跳节流。
- 翻页动画必须可关闭并记住用户偏好；动画保持轻微横向位移、透明度与缩放，不模拟夸张 3D 卷页，并在 `prefers-reduced-motion` 下强制关闭。
- 不新增大面积 banner、说明卡、装饰区来承载功能入口，除非它是用户当前任务的核心界面。
- 阅读器侧栏新增面板时应保持轻量任务面板形态；例如「记忆」这类长期参数编辑入口，应使用紧凑表单、明确保存反馈和安静空状态，不能压过正文阅读和问答主流程。
- 设置页必须保持分区控制台形态：AI 服务、批量配置、数据备份、隐私安全和高级维护这类不同任务不能重新堆回单列长表单。
- 设置页的安全、备份和危险操作说明应以状态摘要、折叠说明或独立维护面板呈现，避免大面积警告卡占据首屏。
- 破坏性操作必须可发现、可确认；普通次级操作应收敛到菜单或线框按钮。
- 新增格式、AI 配置、阅读计划逻辑时，不得顺手改变书架的视觉比例和交互边界。
- 改动共享组件前，先确认它影响哪些页面；不能只看当前功能路径。

## 验收清单

每次改动书架、阅读器、设置页或共享 UI 后，至少检查：

- `npm run build` 通过。
- `git diff --check` 通过。
- 全局背景使用米纸底色与 3%–6% 低透明度纸纹。
- 朱砂色只用于品牌、状态、关键操作、进度和印章点缀。
- 顶部导航当前态明确，但不过度抢眼。
- 书籍卡片逐步采用藏书票结构，包含书封、书名、作者、状态、进度和朱砂章。
- `0%`、`Processing`、`No Data` 等机械状态已替换为自然阅读语气。
- 桌面端封面网格列宽、间距和页面左右起点稳定。
- 封面主点击、菜单点击和 hover/focus overlay 互不抢事件。
- PDF 真实封面、MOBI 文字封面、长书名和英文书名都能保持封面比例。
- 书架顶部标题、日期信息和上传/测试导入操作不拥挤、不漂浮。
- 导读页进入正文的页面级过渡不造成额外滚动感、闪屏、正文延迟挂载或活动记录提前触发。
- 移动端不依赖 hover，内容自然可达且没有横向溢出。
- 新增 UI 文案没有重复解释界面怎么用，能用布局和控件表达的就不写说明。

## 维护规则

- 视觉改动先更新本文件的标准，再记录到 [UI_CHANGELOG.md](./UI_CHANGELOG.md)。
- 如果某次功能开发不得不打破这里的标准，需要在变更说明里写清原因和替代约束。
- 评审 UI 改动时，优先检查布局稳定性和交互边界，再讨论审美细节。
