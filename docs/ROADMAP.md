# 读伴 Roadmap

> 最后更新：2026-07-22

这份文档用于维护「读伴」的产品路线图。它关注项目接下来要往哪里走、哪些事情优先做、每个阶段完成到什么程度算达标。

文档分工见 [README.md](./README.md)。产品和架构共识见 [PROJECT_NOTES.md](./PROJECT_NOTES.md)，统一实施日志见 [APP_EVOLUTION_LOG.md](./APP_EVOLUTION_LOG.md)，P6 生产化历史见已冻结的 [PRODUCTION_UPGRADE_PLAN.md](./PRODUCTION_UPGRADE_PLAN.md)。

## 产品定位

读伴是一个本地优先的桌面 AI 伴读应用，同时保留浏览器体验入口；它不只是 PDF 阅读器。

它的核心价值是：用户上传自己的书籍后，应用帮助用户理解书籍结构、制定阅读节奏，并在阅读前、阅读中、阅读后提供读伴式陪伴。

## 当前状态

当前项目已完成 Tauri 桌面 App 的 P6 生产化工程基线，正在推进 P7 连续陪读与按需协助；主动提问已经从产品路线撤下：

- 用户可上传 PDF 或 MOBI，并在本地提取分页文本；PDF 使用原版页渲染，MOBI 使用文本页阅读。
- 用户可确认书名、作者、章节结构和章节用途。
- 用户可在上传后进入开书设置页，先通过几轮对话设定这本书的读伴，并在最后一轮定下本书读伴侧重点；之后只需要设置阅读节奏、开始日期、每周阅读日和长章节拆分。整本书开书地图作为可选增强项保留。
- 应用可基于章节用途、阅读节奏和读伴侧重点生成本地阅读计划草稿。
- 整本书导读已支持快速导读、完整导读底稿、阅读路线、阅读难点、建议读法和读伴侧重点选项。
- 开书契约上下文已贯穿章节导读、阅读中问答和读后交流，让读伴能把当前内容放回整本书问题、结构位置和用户阅读目标中。
- 阅读器已支持三段式阅读体验：读前导入、正文阅读、读后交流。
- 阅读器正文页已按视觉标准完成第一轮纸面化：正文页、PDF 承托、右侧读伴栏和阅读控制都统一到米纸、淡墨、朱砂点睛的体系。
- 阅读器侧栏头像区已新增「本书读伴」设置入口，支持五类可执行策略、可编辑记忆和仅当前问题生效的单次覆盖，不增加新的侧栏 Tab。
- 本书级 AI 聊天曾完成独立入口和本地历史保存；P7.3 试用后已撤下该入口，旧消息继续作为兼容历史保留，不删除用户记录。
- 从读前导入进入正文时支持克制的纸面翻页过渡，并对 reduced-motion 场景降级。
- 正文阅读会根据书籍格式分流：PDF 默认渲染原版页并叠加可选中文字层，MOBI 使用文本阅读视图。
- 书架正在测试封面优先形态：PDF 优先提取第一页作为封面，MOBI 或失败场景使用文字封面兜底。
- 阅读位置、完成状态、聊天记录、导读、高亮和笔记在浏览器版保存到 IndexedDB；Tauri 桌面版已开始迁移到 SQLite + App 数据目录文件存储，书籍元数据、章节索引、原始文件索引、分页文本、阅读计划、阅读进度、笔记、聊天、读后交流和章节导读缓存已结构化，桌面 API Key 已迁入系统 Keychain。
- AI 调用采用 BYOK 模式，支持 Anthropic Claude 和 OpenAI-compatible Chat Completions；浏览器版仍直连模型服务，Tauri 桌面版已通过 Rust command 代理模型请求。
- 桌面版 AI 使用路径已加入进程内 Keychain 密钥缓存，用于减少同一使用链路中的重复系统授权弹窗；缓存不落盘，保存或删除密钥后会失效。
- 设置页支持从预设 TXT 模板批量导入 AI 供应商、模型、Base URL、API Key 和价格配置，也支持把当前 AI 配置导出为 TXT。
- 已支持手动生成章节导读、当前章节伴读问答、选中文字追问、高亮笔记、笔记管理和读后交流追问。
- 已建立后端开发标准和后续 AI 接手提示词文档，用于约束 Tauri/Rust、SQLite、Keychain、备份、AI transport 和交接流程。
- 已建立产品提示词编写规范，用于约束读伴文风、模板句、背景知识边界和文风版本元数据。
- 已新增 [PRODUCTION_UPGRADE_PLAN.md](./PRODUCTION_UPGRADE_PLAN.md)，将阶段 5 之后的数据可靠、正式发布、安全隐私、诊断、CI、QA、自动更新和 public alpha 拆成 P6.x 生产级升级步骤。

## 路线原则

- 先验证核心伴读体验，再扩展书库和同步能力。
- 保持纯前端 MVP 的迭代速度，同时让存储层和 AI 调用层保持可迁移。
- AI 功能必须围绕阅读动作服务，避免变成泛聊天入口。
- 阅读器要优先保证安静、稳定、可持续阅读，而不是堆满功能按钮。
- 用户自己的 PDF、API Key、笔记和聊天记录默认只保存在本地。

## 阶段路线

### P0：伴读闭环打磨

目标：让用户能稳定完成一次「读前准备 -> 正文阅读 -> 提问/笔记 -> 读后反思」的完整章节阅读。

当前进展：

- 已完成三段式阅读会话、PDF 原版页阅读、选中文字提问、高亮笔记和读后交流追问。
- 已完成每日阅读结束页：读完当天任务后不自动跳到下一章，而是让用户回书架或提前开始下一章。
- 已完成阅读目录入口，支持回顾已读内容和进入未读内容。
- 已完成开书契约上下文接入：章节导读、伴读问答和读后交流都会带入 `wholeBookGuide` 与 `companionFocus`。
- 已完成本书读伴记忆面板，用户可在阅读器中编辑 `readingProfile.companionFocus`。
- 已完成本书级读伴聊天 P0-P2：从书架菜单进入独立聊天页，支持真实流式回答、历史持久化、全书上下文、进度上下文、不默认塞入整本书全文的剧透控制、读伴回答记到笔记和本书聊天清空。

优先事项：

- 验证开书契约在导读、问答和读后交流中的真实效果，避免机械复述开书分析。
- 验证本书级聊天在“回味、接上思路、准备下一次阅读、整理阶段性理解”里的真实效果，避免变成泛聊天入口。
- 继续评估本书聊天里的回味卡片、长期记忆自动摘要和聊天导出，确保沉淀能力服务阅读而不是扩成泛知识库。
- 继续让读后交流更深地带入本阅读项的导读目标、阅读中问答、高亮和笔记。
- 打磨「记忆」面板的入口层级、保存反馈、移动端表现和空状态。
- 优化读前导读的流式生成和失败重试体验。
- 支持清空当前阅读项聊天记录。
- 继续改进高亮、引用、页码来源和笔记详情的稳定性。

完成标准：

- 用户读完一个阅读项后，可以得到有针对性的反馈。
- 用户能清楚知道 AI 回答主要依据哪一页、哪段文字或哪个章节。
- 阅读器主流程不依赖刷新页面或手动修复状态。

### P1：阅读计划智能化

目标：从简单本地计划草稿升级为更贴合书籍内容和用户目标的阅读计划。

当前进展：

- 已完成开书设置页第一步多轮对话化：用户通过“来处、好奇心、陪读方式”几轮回答设定读伴，读伴形态会逐步完善；整本书导读不再压在第一步里。
- 已完成新版 `readingProfile.companionFocus` 设计与保存。
- 已完成本地计划草稿，支持长章节按当前节奏拆分。

优先事项：

- 基于 `wholeBookGuide`、章节结构、用户节奏和读伴侧重点调用 AI 生成整体阅读计划说明。
- 为每个阅读项生成阅读目标、关键问题、难度提示和全书位置。
- 支持用户编辑 AI 计划，并保存最终版本。
- 支持重新生成计划时保留用户已有进度和笔记。

完成标准：

- 阅读计划不只是按章节平铺，而能解释为什么这样安排，并能回应用户开书时写下的阅读意图。
- 用户可以接受、调整或放弃 AI 计划，不会覆盖已有阅读记录。

### P2：阅读体验增强

目标：让读伴成为可以长期阅读的主阅读界面。

当前进展：

- 已支持 PDF 原版页阅读和 MOBI 文本页阅读两条路径。
- 已完成阅读器正文页的米纸视觉底座：题签式顶栏、正文纸页、PDF 页面承托和右侧读伴札记栏。
- 书架已开始从信息卡片转向封面优先的轻量书库视图。
- 导读页到正文页的进入动效已完成多轮性能和节奏打磨。
- 已新增 [UI_DESIGN_STANDARDS.md](./UI_DESIGN_STANDARDS.md)，用于约束后续 UI 改动的视觉比例、交互边界和验收清单。

优先事项：

- 增加字号、行距、页面宽度等阅读设置。
- 增加章节导航和页码跳转。
- 优化长 PDF/MOBI 的加载、滚动和当前页识别性能。
- 改进移动端阅读和 sidebar 使用体验。
- 评估 PDF.js worker、canvas 渲染和 text layer 的按需加载或拆包。

完成标准：

- 常见 PDF 在桌面端阅读流畅。
- 用户能快速跳转章节和回到上次阅读位置。
- 移动端至少可以完成阅读、提问和查看笔记的核心动作。

### P2.5：整理这本书与复盘

目标：让每一本书拥有一个独立的复盘空间，用户可以在这里整理全书笔记、摘录、读伴回答、读后交流和重点。

当前进展：

- 已完成「整理这本书」前端原型，从书架菜单和本书聊天页进入。
- 整理页聚合本书阅读进度、全书笔记、本书聊天和读后交流。
- 页面按「现代数字书斋」标准收敛为左侧本书状态栏 + 右侧主案面标签页，默认聚焦笔记整理。
- 已支持在整理页里筛选、编辑、保存和二次确认删除笔记。
- 「重点」和「复盘」当前基于整本书导读、笔记、聊天和读后交流做轻量前端整理，不新增 AI 调用或 schema。
- 已根据试用反馈把入口文案改为「整理这本书」，并把右上角大胶囊控件收敛成标题下方轻量页签和左对齐筛选片。
- 会客厅视觉已按米纸书案方向继续收敛，弱化后台感，并让「继续阅读」「设定读伴」这些下一步动作更清楚；独立「和读伴聊聊」动作已在 P7.3 反馈收敛中撤下。

优先事项：

- 设计知识卡的数据结构：概念卡、问题卡、论点卡、人物/事件卡、我的理解卡。
- 支持把笔记整理为知识卡，并保留来源链路：原文页码、阅读项、读伴回答或读后交流。
- 支持整理页内的标签、筛选和搜索。
- 评估 AI 自动整理：阶段性复盘、知识点候选、回味卡片和读书报告草稿。
- 保持单书优先，不急于扩展成跨书知识库。

完成标准：

- 用户离开阅读器后，仍能围绕一本书持续整理自己的理解。
- 笔记、读伴回答和知识卡之间能互相追溯来源。
- 整理页服务单本书复盘，不把产品提前推成泛知识库。

### P3：书库可迁移与备份

目标：降低 IndexedDB 作为唯一存储带来的长期风险。

当前进展：

- 已在设置页提供本地备份导出/导入。
- 桌面版备份已升级为目录式 `manifest.json + files/` v3，包含书库、原始文件、分页文本、阅读进度、导读、笔记、聊天和读后交流，默认不包含 API Key。
- 桌面版支持备份清单、导入前预览、校验报告、manifest/file sha256、合并导入、覆盖恢复、失败自动回滚、备份名称/备注、删除和外部路径导入。
- 桌面版 schema 已到 `9`，schema 初始化已收束为显式迁移器入口。

优先事项：

- 为 schema 迁移和备份恢复继续建立固定测试夹具，覆盖旧库、旧备份目录、坏备份和大书库。
- 后续评估 zip/tar 压缩归档、备份签名和原生外部目录选择器。

完成标准：

- 用户可以把关键阅读数据备份到本地文件。
- 清空浏览器数据前，用户有明确的数据迁移路径。

### P4：长期产品形态

目标：决定读伴是继续作为浏览器工具，还是演进为桌面个人书库。

App 化专项路线和每次实施记录维护在 [APP_EVOLUTION_LOG.md](./APP_EVOLUTION_LOG.md)。

可能方向：

- 桌面版使用本地文件系统 + SQLite 管理长期书库。
- 可选云同步只同步用户授权的数据，不默认上传 PDF 原文件。
- 桌面版已引入 Tauri Rust command 作为本地模型请求代理；浏览器版后续仍可评估是否需要后端代理。

完成标准：

- 有明确的长期存储策略。
- 有明确的模型调用策略。
- 浏览器版和桌面版的职责边界清晰。

### P7：连续陪读与按需协助

目标：先把导读、读中和读后串成同一位读伴的连续脉络，让提问、划词、笔记和回想都发生在书籍阅读现场，再建立可靠的内容锚点、按需上下文、策略约束和记忆闭环。读伴只响应用户主动发起的操作，不在阅读过程中自行提问或弹出提醒。

路线原则：

- 先验证连续体验，再冻结底层 schema；避免为尚未验证的交互提前建设错误数据结构。
- 导读、读中和读后共享持续挂载的读伴壳层；不再设置脱离阅读器的独立聊天入口。
- 用户设置分为可验证的硬规则、可编辑的软记忆和单次会话覆盖，不再使用没有功能映射的抽象人格/关注标签。
- 剧透边界必须由未读上下文过滤保障，回答深度必须由输出 token 和结构预算保障，不能只依赖 prompt。
- 阅读状态只用于保存进度、防剧透和恢复上下文，不用于触发主动提问。

实施步骤：

1. **P7.1 陪读脉络契约与前端适配层**：把现有导读、章节问答、读后交流、本书聊天和笔记规范化为统一只读 journey model，暂不迁移 schema。
2. **P7.2 持续挂载的读伴壳层**：建立共享 presence/context/timeline/composer，在导读、正文和读后之间保留草稿、滚动、引用与生成状态。
3. **P7.3 三阶段串联与共享元素动效**：串通导读、读中和读后，完成共享读伴标记和记录卡过渡；阶段中用于试验的主动提问原型已在 2026-07-17 撤下。
4. **P7.4 行为策略与记忆设置收敛**：建立剧透边界、回答深度、追问和知识边界四类可见硬规则，以及软记忆和单次覆盖；旧主动程度字段固定降级为关闭。
5. **P7.5 统一事件、来源与存储迁移**：把已验证的脉络固化为版本化事件、来源锚点、策略、记忆和迁移/备份契约。
6. **P7.6 让读伴准确知道你读到哪里**：统一正文分块与来源定位，区分到过、认真读过和已完成，并让严格防剧透只使用当前页与确认已读内容。已完成。
7. **P7.7 按需上下文编排与缓存**：已完成。合并原 P7.7 与 P7.9；三条书内 AI 路径统一按场景选择可核验材料，严格过滤未读内容，记录来源与失效指纹，并落实回答硬预算、章节导读制品缓存和 usage/费用链路。
8. **P7.8 读伴整体界面与视觉重构**：已完成。界面审计、三案定稿、第二版原型还原、组件与资产重构、页边印记收起/唤醒、宽窄窗口与桌面回归均已完成；第一版被否决资产和重复组件已清理。
9. **P7.9 成果沉淀与用户可控记忆**：已完成。读后时间线、成果索引、记忆撤销、后续导读受控承接、跨章节来源管理、旧记录补全和固定备份回归均已落地。
10. **P7.10 读伴静默状态与动效**：已完成。准备导读、安静陪读、回答中、等待用户、保存记录、完成、错误和离线统一映射到真实任务状态；正文静默态无循环动效，离线可继续阅读，reduced motion 会关闭全部状态动画。
11. **P7.11 诊断、QA 与 Public Alpha 验收**：已完成。设置页可解释每次请求的选材、排除原因、策略、预算和缓存状态；诊断摘要不保存正文、笔记、回答或密钥；七类固定样本、P7 专项预检和候选包人工清单已落地。

2026-07-20 视觉优先级调整：新增 P7.8“读伴整体界面与视觉重构”；原 P7.8-P7.10 依次顺延为 P7.9-P7.11。历史日志中的旧编号继续保留当时语义。

阶段里程碑：

- **P7-A 连续体验基线**：完成 P7.1-P7.4，用现有数据走通导读、读中、读后、共享壳层、动效和设置模型。
- **P7-B 按需行为闭环**：完成 P7.5-P7.7，一本正常文本型 PDF 的阅读项走通“导读线索 -> 阅读 -> 用户提问 -> 回答”，且来源、防剧透、回答预算和缓存由代码保证。
- **P7-C 长期陪伴收口**：完成 P7.8-P7.11，先统一读伴整体界面与视觉，再形成跨章节记忆、静默状态、诊断、迁移和正式候选回归。

完成标准：

- 用户在导读、正文和读后之间能感知到同一位读伴与同一条陪读脉络，场景切换不丢状态。
- 每个硬设置都能改变明确模块并由自动化测试验证；软记忆可查看、修正和删除。
- 严格不剧透时未读正文不会进入请求上下文；简练/均衡/深入有稳定的硬预算差异。
- 用户主动发起的问答能说明使用了哪一页、哪一段，未读内容不会被带入。
- 用户回答进入本节记录，并能在用户可控前提下影响后续导读、按需回答或记忆。
- 读伴在开书、导读、读中和读后保持同一视觉身份与界面逻辑；各场景自然衔接，不像关闭一个功能后再打开另一个面板。
- 解析、AI、缓存或动效失败时，阅读流程保持完整，不伪造内容、不泄露正文、不阻断操作。

完整设计、字段边界、依赖和逐步验收见 [COMPANION_ACTIVE_READING_PLAN.md](./COMPANION_ACTIVE_READING_PLAN.md)。

### P8：手机版 App

目标：在不依赖云后端的前提下，把读伴扩展为适合手机持续阅读的本地优先 App。首期优先验证 iOS，保留 Android 共享实现路径。

实施步骤：

1. **P8.1 移动技术验证**：评估 Tauri mobile 复用现有 React、存储适配和 AI transport 的可行性，确定 iOS/Android 壳层边界。
2. **P8.2 移动信息架构**：重排书架、阅读器、读伴、笔记和设置导航，适配安全区、单手操作、横竖屏与系统返回手势。
3. **P8.3 文件导入与本地存储**：接入系统 Files/分享入口、移动沙盒目录和安全密钥存储，保持书籍默认不上传云端。
4. **P8.4 手机阅读器**：完成 PDF 翻页/滚动、手势、窄屏读伴、选区、高亮、笔记和阅读进度恢复。
5. **P8.5 移动 AI 与后台行为**：接入移动端 Keychain/Keystore、请求取消、网络切换、锁屏/切后台恢复和费用保护。
6. **P8.6 本地数据迁移**：在 P9 云同步前，提供受控备份导入/导出和桌面到手机的手动迁移路径。
7. **P8.7 移动 QA 与性能**：覆盖不同屏幕、内存压力、大 PDF、无网、深浅色、动态字体和辅助功能。
8. **P8.8 移动 Beta 分发**：完成移动签名、TestFlight/对应测试通道、隐私说明、崩溃诊断和发布清单。

完成标准：

- 用户可以在手机上独立完成导入、阅读、向读伴提问、记笔记和恢复进度。
- 手机版即使没有账号和云服务也可完整使用。
- 桌面与手机共享核心数据语义，但不强行共享桌面布局和交互。

详细规划见 [MOBILE_APP_PLAN.md](./MOBILE_APP_PLAN.md)。

### P9：云后端与多设备同步

目标：在桌面端、按需陪读和手机版都形成稳定本地闭环后，再引入账号、同步和可选云能力，同时保持本地优先和用户授权原则。

实施步骤：

1. **P9.1 云端需求与数据流决策**：明确多设备同步、托管备份、账号和模型代理各自解决的问题，禁止用一个模糊“后端”同时承载全部需求。
2. **P9.2 账号与设备身份**：设计登录、设备注册、会话撤销、账号删除和无账号继续使用边界。
3. **P9.3 本地优先同步协议**：为书籍元数据、进度、笔记、陪读记录和设置设计增量同步、冲突解决、幂等和离线队列。
4. **P9.4 加密与隐私模型**：决定端到端加密、服务端加密、恢复密钥、设备丢失和审计日志边界。
5. **P9.5 后端基础设施**：建设 API、数据库、对象存储、任务队列、鉴权、限流、监控和备份恢复。
6. **P9.6 分阶段数据同步**：先同步小型结构化数据；PDF 原文件默认不同步，只有用户明确授权后才进入对象存储。
7. **P9.7 可选云能力**：分别评估云备份和模型代理，不让 BYOK、本地使用或基础阅读被云服务锁定。
8. **P9.8 可靠性、合规与 Beta**：完成数据导出/删除、隐私政策、成本控制、灾难恢复、灰度发布和跨设备回归。

完成标准：

- 无账号用户仍可使用本地核心功能。
- 多设备同步可以离线工作、处理冲突，并能解释每类数据是否上传。
- PDF、API Key 和私人阅读内容不会在用户不知情时进入云端。

详细规划见 [CLOUD_BACKEND_PLAN.md](./CLOUD_BACKEND_PLAN.md)。
