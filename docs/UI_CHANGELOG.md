## 2026-07-22 完成页默认成果与屏幕适配收尾

- 完成页默认展开第一个有内容的成果分类，优先顺序为“回答、笔记、读后”；用户手动收起或切换后不会被自动抢回。
- 宽屏采用成果与目录双栏，目录独立滚动；900px 以下改为单列内部滚动，矮窗口使用紧凑间距。
- 1280×720、760×620、480×620 实测均无全局滚动条或横向溢出，主要操作和默认回答内容可正常到达。

## 2026-07-21 读后记录时间线与便签区分

- 移除“读完后，留一个判断”大标题，读后页收束为紧凑的“读后记录”工作区，不再重复解释已经发生的阅读动作。
- 读中的导读线索、提问、回答和笔记沿同一时间线进入读后；场景切换复用时间线共享元素，视觉上像原记录移动并继续展开。
- 页面固定为一个视口，顶部身份区与底部输入区固定，只有中间记录区滚动。进入时默认在最新一条，窗口缩放后继续贴底。
- 输入提示改为“还有什么想聊的，可以接着聊。”，强调这是已有交流的延续。
- 增加“整理本节总结 / 重新整理总结”，总结以居中记录呈现；继续交流时旧总结先撤下，避免和新对话混排。
- 问题继续靠右、读伴回答靠左；笔记改为窄幅暖纸便签，使用标题分隔、轻折角和纸张阴影，严格区别于聊天气泡。
- 1280×720 与 760×620 浏览器实测无全局滚动条，时间线默认和缩放后均位于底部；P7 全量测试通过。
- 完成页的成果入口现在默认展开第一个有内容的类别，顺序为“回答、笔记、读后”；用户手动切换或收起后保持其选择。
- “今天这段读完了”完成页改为单视口工作区：宽屏左侧呈现完成概览、成果和下一步操作，右侧目录独立滚动；窄屏改为内部滚动的单列布局，不再出现浏览器或 WebView 的全局滚动条。

## 2026-07-21 P7.9.1 本节记录与记忆确认

- 阅读完成页以「本节留下了什么」替换只读摘要，提供“我留下的理解”和“还没有想通的问题”两个可编辑区域，并显示本节提问、笔记和用户回想计数。
- 保存、删除和“确认让读伴记住”是独立动作。保存不会自动修改长期记忆；已确认内容发生编辑后，需要用户再次确认才会更新。
- 只有用户自己的回想、真实笔记或手工输入可以形成记录；模型回答、`AI 回答` 等通用标题和旧兜底文案不会生成空洞成果。
- 删除本节记录不会暗中删除已经确认的本书记忆；长期记忆继续在本书读伴设置中统一管理。
- 宽屏使用双栏编辑，窄屏自动改为单栏；按钮、状态和错误提示沿用朱砂批注界面，不增加新的独立弹层或嵌套卡片。
- P7 全量测试、formal/test build、安全扫描、Rust 测试和 Test.app 签名校验通过；最终测试包可执行文件 SHA-256 为 `f22fbe936d4e6a230e453e878ad667d63e17f22352777d3b083a250ba109ba49`。
- 真实桌面回归已确认旧错误记录自动变为空白、输入后保存和记忆确认按钮正确启用；保存后的重启恢复点击因 macOS 锁屏未完成，不能记为人工通过。

### 后续体验修正

- 统计项改为可点击展开：问答会同时展示问题和对应读伴回答，笔记展示本节保存内容，“回想”改为含义明确的“读后回答”。数量为零时入口保持禁用。
- 两个并排 textarea 改为纵向对话：读伴问题靠左、用户回答框靠右，减少表单感，并在窄窗口继续保持单列完整宽度。
- 已确认记忆新增“撤销记忆”。撤销会移除本书读伴记忆并清除本节关联，但保留本节记录；旧异常来源记忆也可以从对应完成页撤销。
- 最新 Test.app 已真实验证三类入口展开/切换、对话式布局和撤销按钮条件显示；签名校验通过，可执行文件 SHA-256 为 `b17f660d86a35a2b816b231d70805a1cab6b0ba1bb4222407337a743795ef614`。

### 最终收敛

- 读后聊天已经承担用户表达和追问，完成页再次要求填写理解与问题会造成重复和场景断裂，因此移除两轮输入、“保存本节记录”和“确认让读伴记住”。
- 完成页最终只保留“回答 / 笔记 / 读后”三个入口，点击展开已有内容；回答数量按实际读伴回答计算，展开时同时保留对应用户问题。
- 只有存在本节关联长期记忆时才显示紧凑的记忆行和“撤销记忆”，没有记忆时不占据页面空间。
- 最终 Test.app 已确认空成果页不出现输入框、保存或确认按钮，三个零计数入口正确禁用；P7 全量测试、formal/test build、安全扫描、Rust 测试和签名校验通过。可执行文件 SHA-256 为 `02e6f5a41d8d440605e6e7b8da2ca1891458f7cd76efb59df55b2390cf7cd6cf`。

## 2026-07-21 桌面阅读页闪烁修复

- 桌面版在读伴开合或场景承接后偶发书页闪烁，滚轮触发重绘后恢复。原因是 macOS Tauri WebView 对长正文或 PDF Canvas 的整页 View Transition 快照合成不稳定。
- Tauri 桌面端不再调用原生整页 View Transition，改用已有的局部读伴淡入与位移动画；网页端在支持时继续使用原生共享元素过渡。
- 修复不改变读伴开合状态、阅读页码、草稿、时间线、PDF/MOBI 渲染和减少动态逻辑。
- 新增运行时测试，明确锁定“Tauri 禁用整页快照、普通浏览器保留原生过渡、无 API 时使用局部降级”三种情况。
- 新 Test.app 已在 MOBI 滚动/翻页两种模式下完成读伴收起、展开和翻页回归，书页无需滚轮触发重绘即可保持可见；测试包 SHA-256 为 `921c15499665b03a625f27bc40c41561ac6ff0168411e05f583a42abbcffd744`。

## 2026-07-20 本书读伴设置通俗化

### 需求背景

- “只给方向，不说答案”“知识范围”“回答后的追问”等表述需要用户先理解产品术语，仍然不知道选择后读伴会具体怎么回答。

### 本次更新

- 弹层标题改为“读伴怎么回答你”，并说明设置只对当前书生效。
- 四组字段改为“能不能聊到后文、回答要讲多细、回答后还要问你吗、回答时可以参考什么”。
- 每个选项下方显示当前选择的实际效果。例如“提醒我留意线索，但不说后文”会明确说明读伴可以提醒该留意什么，但不会透露后续事件或最终答案。
- 本书记忆改为“希望读伴记住的事”，补充适用范围和“遇到生词先用白话解释”的示例。
- 输入区“仅本次”改为“这次回答”，选项统一使用“这次简短一点”等口语表达。

### 兼容

- 只修改文案与说明，策略值、默认项、保存格式、防剧透逻辑和已有书籍设置均保持不变。
- 专项测试、测试构建、安全扫描与桌面 Test.app 重建通过；真实阅读页两列弹层中长选项和说明均完整显示，无重叠或截断。
- 试用后将字段标题“回答后还要问你吗”调整为“回答后是否需要追问”，选项和行为不变。
- 再次试用后去除过度口语化表达：弹层改为“本书读伴设置”，选项使用“简要 / 标准 / 详细”“不追问 / 信息不足时追问 / 每次回答后追问”等正式但直观的表述。
- “信息不足时追问”不再使用模糊的“有帮助”判断：只有问题含义不明确或缺少必要信息时才提出一个澄清问题，信息足够时直接结束回答。
- “回答时可以参考什么”收敛为“回答参考范围”；本组文案以中性、准确和可理解为准，不再使用聊天式口吻。
- 桌面测试包实测四组标题、当前选项与说明均完整显示，两列布局无截断或重叠。

## 2026-07-19 测试包 Keychain 失败提示

### 需求背景

- 本地 `读伴 Test.app` 重建后可能失去旧 Keychain 条目的读取授权，界面只显示“读取系统 Keychain 失败”，用户不知道下一步该做什么。

### 本次更新

- 错误提示改为引导用户打开设置，重新填写并保存当前模型的 API Key。
- 测试包构建会在本机证书可用时使用稳定 Developer ID 签名，减少后续重建再次失去授权的情况；正式版和测试版继续使用不同 bundle id 与 Keychain service。
- 不在设置输入框中回显密钥，不自动删除或迁移正式版密钥。

### 当前验证

- 脚本语法、测试前端构建、Rust 定向测试和差异检查通过。
- 首次 Developer ID 测试包签名被锁定的登录 Keychain 阻止，返回 `errSecInternalComponent`；解锁后需重建并核对 Team ID，再由用户在设置中重新保存一次测试 Key。
- 为不中断界面验收，已用临时签名重建并重启最终 Test.app；签名完整性、测试 bundle id 和 schema 10 启动日志均通过。此包仍可能无法读取旧 Keychain 条目。

- 第一版模板仍需要自己填写或调整较多字段，实际使用时不够省心。

### 本次更新

- 设置页新增「AI 批量配置」区域，支持上传 TXT 配置文档。
- 上传后会解析 `key = value` 或 `key: value` 格式，并立即合并保存到本地 IndexedDB。
- 新增「下载模板」入口，模板文件为 `public/ai-config-template.txt`。
- 模板预填 Anthropic、OpenAI、DeepSeek、Kimi 的常用 Base URL、模型名和可用价格字段，用户主要只需要粘贴 API Key。
- TXT 支持 `[anthropic]`、`[openai]`、`[deepseek]`、`[kimi]`、`[openai-compatible]` 分组，也支持完整 key，例如 `openaiCompatible.baseUrl`。
- 只填一个 OpenAI-compatible 供应商的 API Key 时，会自动选择该供应商并写入对应 Base URL、模型和价格。
- 导入时只覆盖文档中填写了值的字段；空值、不认识的字段和格式错误的行会跳过。
- 新增「下载当前配置」按钮，可把当前表单里的 AI 配置导出为可重新导入的 TXT。

### 兼容与限制

- 该功能仍是纯前端 BYOK，不引入后端，也不会把 API Key 上传到项目服务器。
- 当前仍是 BYOK 配置导入，不导入多套 API Key；P6.4.6 起支持 `[profiles]` 导入任务级非敏感模型 profile，如果同时填写多个 OpenAI-compatible API Key，仍需要用 `provider` 指定默认供应商。
- 导出的当前配置会包含 API Key，只适合保存在用户可信的位置。

## 2026-06-08 增加 MOBI 书籍导入与文本阅读

### 需求背景

- 当前书架只支持 PDF 上传，用户希望可以导入 MOBI 格式书籍。
- 现有阅读器以 PDF 原版页渲染为主；MOBI 没有可直接复用的 PDF canvas/text layer，需要走文本阅读路径。
- 书籍信息、阅读计划、进度、笔记和问答都依赖页码字段，需要兼容 MOBI 的文本分页。

### 本次更新

- 新增 `src/lib/bookFormats.js`，统一识别 PDF/MOBI 文件格式、上传 accept 配置和页码单位展示。
- 新增 `src/lib/mobi.js`，接入 `@lingo-reader/mobi-parser`，支持在浏览器端解析 MOBI/KF8，读取 metadata、toc/spine 和章节 HTML。
- MOBI 正文会清洗为可阅读文本，并按约 2200 字符切成文本页，继续复用现有章节、阅读计划、当前页上下文和笔记数据结构。
- 新增 `src/components/TextBookReader.jsx`，为 MOBI 提供文本阅读视图，支持当前文本页识别、恢复上次位置、划选原文后问读伴或添加笔记。
- 书架上传入口改为支持 PDF 或 MOBI；保存逻辑从 `createBookFromPdf` 扩展为通用 `createBookFromParsedFile`，同时保留旧函数兼容。
- 阅读器根据书籍格式分流：PDF 继续使用 `PdfReader` 原版页渲染，MOBI 使用 `TextBookReader` 文本阅读。
- 书架、书籍信息页、开书计划页、阅读目录、聊天提示和笔记详情中的页码文案改为按格式显示“页”或“文本页”。
- 删除书籍确认文案改为“原始文件”，避免只描述 PDF。

### 兼容与限制

- 已有 PDF 书籍如果没有 `format` 字段，会按文件名回推，默认仍作为 PDF 处理。
- MOBI 当前提供文本阅读和伴读能力，不渲染 Kindle 原版版式，也不会显示 MOBI 内嵌图片。
- MOBI 章节优先使用 TOC 映射；TOC 不完整时按 spine 顺序生成章节，用户仍可在书籍信息页手动调整。
- `@lingo-reader/mobi-parser` 作为动态导入依赖，只在上传 MOBI 时加载，避免直接增加首屏主包。

## 2026-06-08 书架页小清新改版

### 需求背景

- 书架页需要更像真实书架，气质要贴近「读伴」手写 logo 的文艺、小清新感觉。
- 书籍操作不应全部平铺在卡片底部，适合收进菜单或下拉框。
- 书籍需要支持删除，并明确清理本地保存的 PDF、阅读计划、进度、笔记和问答记录。
- 第一版书架仍偏厚重，顶部「读伴书架」banner 信息重复，详情区域占用空间太多。
- 书籍详情适合改成鼠标悬浮或键盘聚焦后展开，让默认书架更轻、更安静。

### 本次更新

- 移除大尺寸书架 banner，改为轻量页面标题栏，上传入口保留在右侧。
- 开发测试书入口从独立横幅移到标题栏按钮区，减少页面冗余。
- 书籍卡片默认只展示书脊、状态、标题、作者、阅读概览和主操作按钮。
- 页数、章节、识别方式、章节角色统计和完整阅读进度改为 hover/focus 展开。
- 书籍次级操作统一放入右上角菜单：阅读目录、书籍信息、开书设置、删除书籍。
- 书脊与按钮使用更柔和的鼠尾草绿、浅蓝、浅紫、浅棕等颜色，降低木质层板的厚重感。
- 删除书籍时增加确认提示，并清理当前书籍相关的本地存储数据。

### 后续修正

- 进一步移除标题区域的面板感，不再显示 Local Library、大背景或明显分隔线。
- 书卡详情展开从纯 CSS hover 改为单本书状态控制，避免鼠标悬浮时多本书同时展开。
- 书架网格改为顶部对齐，避免同一行卡片被展开卡片撑高。
- 进一步执行 Flat 2.0 方向：移除木质托盘和立体书脊，书籍视觉改为 2D 平面封面。
- 定义偏灰紫蓝的全局 Primary，仅用于书卡底部的核心阅读按钮；上传和测试导入改为次级线框按钮。
- 放大数据面板内边距，增强标题、作者、标签和数据之间的字号/字重层级。
- 统一书卡默认高度，避免长短标题导致卡片尺寸参差；悬浮展开时仅当前卡片高度自适应。
- 建立克制的书卷感字体栈，整体靠近 logo 气质，但不使用花哨手写字体。
- 将主色从冷灰紫回调为温暖焦糖棕，封面、状态、进度等辅助色统一到暖杏、橄榄、陶土色系。
- 固定书卡顶部封面/标题信息区高度，让阅读概览与底部主按钮在多卡片之间保持对齐。
- 撤掉小封面装饰方案，回到以书名、作者、状态、页数为核心的信息卡；桌面恢复两列宽卡比例。
- 页面最大宽度收回到 `max-w-5xl`，避免卡片在宽屏中过窄或漂浮。
- 去掉展开详情里的第二条阅读进度条，只保留默认态的阅读概览，展开后展示补充数据和打卡信息。
- 展开动画改为固定卡片高度过渡 + 详情区域 grid 行展开，减少 `max-height` 动画造成的顿挫。
- 书卡展开增加轻量下探提示符，并为 hover 展开加入短暂意图延迟；去掉整卡上浮位移，让展开从“突然弹开”改为更平缓的内容铺开。
- 修复卡片固定高度过小导致底部阅读按钮脱离卡片的问题；按钮改为卡片底部固定操作区，默认态和展开态重新分配高度与底部留白。

### 设计原则

- 页面顶部只做导航和操作，不再重复品牌表达。
- 默认状态优先呈现书本本身，详情只在用户表达兴趣时展开。
- 操作入口保持克制，但删除等破坏性操作必须可发现、可确认。
- 桌面端用 hover 展开，移动端因为没有稳定 hover，详情保持展开，避免信息不可达。
