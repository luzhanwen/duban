## 一、P7 要解决的真正问题

读伴的核心不是让用户选择一个名字、颜色或抽象人格，而是让用户感到：自己为这本书设定的同一位读伴，从开书、导读、阅读到读后始终在场，并且它的行为确实受用户选择影响。

当前产品已经分别具备导读、阅读中问答、读后交流、历史本书聊天、笔记和记忆，但它们仍像几个相邻功能：

1. 场景之间缺少统一的陪读脉络，切换后上下文和视觉关系容易断开。
2. “温和型”“追问型”“概念/论证/背景/应用”等描述没有稳定、一一对应的系统行为。
3. 阅读状态和内容锚点需要服务于防剧透、恢复位置和用户主动问答，不能反过来制造阅读打扰。
4. 防剧透、回答简练程度等关键体验不能只依靠 prompt，必须由上下文过滤和输出预算共同保证。
5. 现有记录分散在导读、聊天、读后交流和笔记中，尚未形成一本书的连续陪读记录。

因此，P7 的顺序调整为：

```text
先串起用户可感知的连续体验
  -> 再建立统一数据与来源契约
  -> 再完善按需上下文筛选和缓存
  -> 最后完成记忆沉淀、视觉身份、诊断和发布验收
```

## 二、不可违反的产品规则

- **同一位读伴**：导读、读中和读后使用同一身份、同一脉络和可追溯记录，不把读伴拆成独立聊天目的地。
- **设置必须生效**：展示给用户的选项必须映射到上下文筛选、输出预算、追问次数、触发频率或知识边界等真实模块。
- **软偏好不是开关**：用户的阅读目标、兴趣、薄弱点和个人背景进入可编辑记忆，不强行压缩成互斥标签。
- **用户先发起**：读伴只响应用户点击导读、提问、记笔记或开始读后回想，不主动弹问或制造新的阅读任务。
- **原文有据**：用户主动问答和导读材料必须绑定书籍、阅读项、正文位置和可核验来源。
- **防剧透靠数据边界**：严格不剧透时，未读正文和含后文信息的整书字段不得进入模型上下文，prompt 只作第二道约束。
- **回答长度靠预算**：简练/均衡/深入必须映射到输出 token、结构和追问限制，不能只修改一句提示词。
- **状态连续**：场景切换时保留输入草稿、滚动位置、生成状态和当前引用，不因动效重新挂载核心会话。
- **失败静默降级**：AI、解析、埋点或动效失败不能阻断阅读；用户仍可正常翻页、滚动、记笔记和完成阅读。
- **隐私本地优先**：阅读事件、陪读记录和记忆默认保存在本地；诊断不记录正文、回答全文或敏感配置。

## 三、目标用户体验

P7 不是新增四个平级 Tab，而是建立一本书的连续「陪读脉络」：

```text
开书设定
  -> 本节导读
  -> 带着“本节线索”进入正文
  -> 用户提问 / 划词 / 笔记
  -> 本节读后交流
  -> 形成“本节留下了什么”
  -> 下一节导读承接此前理解
```

### 3.1 导读

- 导读只留下少量可带入正文的「本节线索」，不堆成长报告。
- 进入正文时，导读中的读伴标记自然缩小并移动到阅读侧栏或页边位置。
- 本节线索在阅读中保持可见、可收起、可引用，但不占用大块正文空间。

### 3.2 读中

- 读伴侧栏展示同一条时间脉络：本节线索、用户问题、划词提问、读伴回答和保存的理解。
- 提示、笔记和阅读项仍是工具视图，不与陪读对话争夺主脉络。
- 专注阅读、窄窗口、滚动和翻页模式都不出现主动打扰。

### 3.3 读后

- 点击「我读完了」后，当前读伴面板扩展为本节回想，不清空读中上下文。
- 回想完成后进入「本节留下了什么」，只聚合已经发生的回答、笔记和读后交流，不要求用户再填写一次总结。
- 没有真实问答、笔记或理解时，不生成空洞总结。

### 3.4 跨章节承接

- 自由提问留在正文侧栏、划词入口和读后交流中，不另设脱离书籍的聊天页。
- 用户可以明确引用某条导读、读中问答、笔记或读后记录；系统不默认把全部历史塞给模型。
- 下一节导读只受控承接相关记录，历史本书聊天作为兼容数据保留，但不再提供独立入口。

## 四、读伴设置模型

### 4.1 可验证的硬规则 `companionPolicy`

| 用户可见设置 | 建议值 | 必须影响的系统模块 |
| --- | --- | --- |
| 剧透边界 | 严格不剧透 / 只给提示 / 开放讨论 | 未读上下文过滤、整书字段过滤、引用范围和 prompt |
| 回答深度 | 简练 / 均衡 / 深入 | 最大输出 token、段落/结构上限、例子与背景预算 |
| 追问方式 | 只回答 / 必要时追问 / 启发式追问 | 最大追问次数、追问触发条件和会话状态 |
| 知识边界 | 以原文为主 / 必要补充 / 允许延展 | 检索来源、外部知识占比和回答中的来源标识 |

硬规则需要版本号、默认值和自动化测试。任何新增选项必须先说明它改变哪个模块，再进入设置页。

### 4.2 可编辑的软记忆 `companionMemory`

以下内容不做互斥开关，而作为用户可查看、补充、修正和删除的记忆：

- 为什么读这本书、希望得到什么。
- 已有知识和容易卡住的地方。
- 感兴趣的人物、主题、案例或现实问题。
- 用户曾经形成的重要判断和仍未解决的问题。
- 用户希望如何把内容用于工作、学习或表达。

记忆只在相关时按摘要和索引取用，不能把全部历史原文注入每次调用。

### 4.3 单次覆盖 `sessionOverride`

用户可在当前问题或当前会话临时指定「这次可以剧透」「这次详细讲」「只基于原文」等覆盖项。覆盖项必须明确显示作用范围，结束后自动恢复本书默认策略。

## 五、核心前端结构

P7 前端围绕一个持续挂载的壳层组织：

```text
CompanionShell
├── CompanionPresence   同一个读伴标记与状态
├── CompanionContext    本书、本节、来源与本节线索
├── CompanionTimeline   导读、读中、读后的统一记录视图
└── CompanionComposer   输入、引用、发送、取消和单次覆盖
```

`CompanionShell` 在导读、正文和读后之间保持挂载，共享元素过渡只服务于书内场景交接，不再承担阅读器与独立聊天页之间的导航。

动效规则：

- 只使用位移、缩放和透明度表达同一元素的移动，避免大面积 blur 和布局抖动。
- 主过渡建议为 320-420ms，文字内容延后 80-120ms；以实测帧率和手感为准。
- 动画必须可中断，快速返回或连续点击不能卡死在中间态。
- 同一时刻只保留一个主运动对象；正文阅读中不使用循环脉冲、闪烁或无意义漂浮。
- `prefers-reduced-motion` 下直接交接状态或使用极短淡入淡出。
- 动画前后必须保持输入草稿、滚动位置、引用、AI 流式生成和请求取消能力。

### P7.1 陪读脉络契约与前端适配层

状态：**已完成（2026-07-14）**。

目标：先用现有导读、聊天、读后和笔记数据建立统一只读视图，不立即迁移存储。

实施内容：

- 盘点现有 `reading guide`、章节聊天、读后交流、本书聊天、笔记和阅读进度的 id 与关联关系。
- 定义前端 `CompanionJourneyItem`：`id`、`bookId`、`readingItemId`、`scene`、`type`、`status`、`createdAt`、`sourceRef`、`payloadRef`。
- 建立 adapter，把现有多份 store 规范化为按时间排序的陪读脉络。
- 定义稳定类型：`guide_clue`、`user_question`、`selection_question`、`companion_answer`、`intervention`、`note`、`reflection`、`session_record`、`book_chat`。
- 明确重复消息、缺少阅读项、旧数据和已删除来源的降级显示。

验收标准：

- 不修改现有 schema，也能按一本书和一个阅读项聚合已有记录。
- 旧书、空书和只有部分记录的书均不报错。
- adapter 有单元测试，排序和去重结果稳定。

实现结果：

- 新增 `src/lib/companionJourney.js`，定义 scene/type/status 常量、稳定 `CompanionJourneyItem`、规范化构建器和筛选函数。
- 新增 `src/lib/companionJourneyStore.js`，只读聚合现有 `bookQuestions`、`bookChat`、`bookReflection` 和 `bookNotes`；不创建新 key、不修改 schema。
- 当前 journey 字段包括 `id`、`bookId`、`readingItemId`、`itemKey`、`itemIndex`、`itemTitle`、`scene`、`type`、`status`、`createdAt`、`updatedAt`、`sourceRef`、`payloadRef` 和轻量 `payload`。
- scene 已统一为 `guide / reading / reflection / book`；type 已覆盖 `guide_clue / user_question / selection_question / companion_answer / note / reflection / book_chat`，并为后续 `intervention / session_record` 预留契约。
- journey id 由存储来源、`itemKey` 和原 source id 生成；没有 source id 时使用内容与时间指纹，重复读取保持稳定。
- 同 source id 的残缺副本按来源去重；旧数组格式和找不到阅读项的记录保留为 `orphaned`，不会被错误绑定到当前章节。
- 导读不复制 `raw`；旧纯文本导读保留为 notes 降级内容。划词问答与笔记保留页码、摘录和规范化矩形来源。
- 新增 `npm run test:companion-journey`，覆盖时间排序、稳定 id、跨类型来源去重、筛选、划词来源、旧 guide、旧数组和失效阅读项。

验证结果：

- `npm run test:companion-journey` 通过，共检查 11 条固定记录。
- `npm run build` 通过；仍只有既有 Vite chunk 体积提示。
- `npm run security:scan` 通过。
- `node --check` 与只读 store 独立 import 检查通过。

- 检查图标、品牌字标、开屏与陪读形象在桌面包中的一致性。
- 静默阅读状态不闪烁、不呼吸提示、不出现未请求的徽标或红点。

验收标准：

- 每种状态都能追溯到真实系统状态，不出现随机表情切换。
- 动效不增加阅读打扰，不造成明显掉帧或输入延迟。
- reduced motion 和低性能环境功能完整。

实现结果：

- 新增集中式八状态契约和联网监听，Reader 的导读、问答、读后、笔记保存和完成场景统一接入 `CompanionShell`。
- `quiet` 和 `waiting` 不启动循环动画；`preparing` / `answering` 只在请求存续期间显示轻反馈，`recording` / `complete` 只播放一次，`error` / `offline` 静态降级。
- 离线提示不阻断阅读；取消请求会随真实 loading 结束退出状态。reduced motion 会关闭全部状态动画与过渡。
- 新增 `test:companion-visual-state` 并纳入 P7 回归；不新增数据库字段、存储 key 或备份版本。
- Test.app 重建确认三种读伴资产和 bundle 图标进入桌面包；正式签名、公证与真实断网切换进入 P7.11 候选验收。

### P7.11 诊断、QA 与 Public Alpha 验收

目标：能够解释用户发起的请求使用了哪些材料、策略和预算，并以一版可回归的候选包完成 P7。

实施内容：

- 开发态查看 journey item、内容锚点、确认已读范围、上下文选材、缓存命中/失效、策略结果和 token 预算。
- 诊断导出只包含 id、类型、状态、耗时、计数和脱敏错误，不含正文、回答全文、笔记或密钥。
- 建立正常文本 PDF、无目录 PDF、扫描页、长章节、旧书、MOBI 和窄窗口固定样本。

- 执行前端测试、Rust 测试、formal build、release preflight、安全扫描和真实 Tauri Test.app smoke test。
- 生成 P7 release checklist 与人工验收清单。

验收标准：

- 测试人员可解释“这次请求用了哪些来源、为什么选中或排除、是否命中缓存”。
- 固定样本可稳定复现并验证策略、缓存和迁移行为。

## 八、范围边界

P7 首轮承诺：

- 文本提取质量合格的 PDF 完成全部按需陪读闭环。

P7 不包含：

- OCR、扫描 PDF、图片/表格视觉理解。
- P8 手机版适配。
- P9 账号、云同步、多人或云端模型代理。
- 任意外观编辑器、虚拟宠物养成或没有真实行为差异的人格系统。
