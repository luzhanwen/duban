# P7 主动陪读引擎实施计划

> 建立日期：2026-07-13
> 状态：已规划，尚未开始实施

本文是 P7 的专项需求与实施来源。路线摘要维护在 [ROADMAP.md](./ROADMAP.md)，每次实际开发完成后仍需同步更新 [PROJECT_NOTES.md](./PROJECT_NOTES.md) 和对应专项文档。

## 一、为什么调整方向

“用户为一本书亲手设定一位读伴，并由它陪完整本书”的产品核心保留，但当前实现存在两个基础问题：

1. 名字、颜色、表情以及“温和型”“追问型”等描述，没有稳定映射到不同的阅读功能。
2. 读伴目前主要响应用户主动提问，缺少章节内容埋点、阅读事件和介入策略，因此无法可靠地主动陪读。

因此，P7 不先增加人格标签或外观编辑器，而是先完成主动陪读的底层闭环。后续人格只能由真实行为差异表现，例如主动频率、问题类型、追问方式和记忆使用方式。

## 二、不可违反的产品规则

- **能力优先**：设置项必须改变真实系统行为，否则不展示给用户。
- **原文有据**：主动问题必须绑定章节、原书页码和正文来源。
- **时机有据**：问题出现必须由内容位置和阅读事件共同触发。
- **默认克制**：每个阅读项只准备少量高价值问题，设置冷却和数量上限。
- **用户可控**：支持关闭、稍后、忽略和调整主动程度。
- **回答有去处**：用户回答必须进入章节记录、笔记或后续上下文。
- **失败静默降级**：文本质量不足、AI 失败或离线时，不能阻断正常阅读。
- **隐私本地优先**：阅读事件默认只存本地；诊断不得记录正文、回答全文或敏感配置。

## 三、核心数据链路

```text
章节/阅读项
  -> 正文块与来源定位
  -> 候选提问埋点
  -> 用户阅读事件
  -> 介入调度器
  -> 阅读中问题
  -> 用户回答/忽略/稍后
  -> 本节陪读记录与本书记忆
```

每个环节都必须可以单独检查。不能只看到一个问题弹出，却无法判断它来自哪里、为什么出现、是否出现过以及用户回答去了哪里。

## 四、编号实施步骤

### P7.1 章节内容地图

目标：为主动陪读建立可靠的章节和原文定位底座。

实施内容：

- 统一 `book -> chapter -> readingItem -> contentBlock` 的关系。
- 每个正文块保存稳定 id、文本摘要、阅读项内位置、原书页码和字符范围。
- 继续区分“本节第几页”和“原书第几页”，底层引用始终使用原书页码。
- 记录文本提取质量、空白比例和是否疑似扫描页。
- 为旧书建立惰性补全策略，不要求用户重新导入书籍。

验收标准：

- 任意正常文本型 PDF 阅读项可以输出有序正文块及可靠来源。
- 从正文块可以回到原书页码和阅读器位置。
- 扫描页或空文本页会被标记为不可用于主动提问。

### P7.2 提问埋点预生成

目标：在阅读前准备少量有明确用途和来源的问题。

实施内容：

- 在进入阅读项前或后台空闲时，为本节生成候选埋点。
- 首期问题用途只使用有明确功能的类别：理解检查、论证辨析、前后联系、个人反思。
- 每个埋点保存问题、用途、触发正文块、最早/最晚触发位置、来源摘录和来源页码。
- 保存模型、prompt 版本、内容指纹、生成状态和费用摘要，支持缓存失效。
- 每个阅读项默认生成 3-5 个候选，不代表都会展示。

验收标准：

- 问题不能脱离原文，来源页码和摘录可以人工核验。
- 同一内容和 prompt 版本不会重复付费生成。
- 生成失败时阅读器正常工作，并允许稍后重试。

### P7.3 阅读事件采集

目标：知道用户是否真正到达、停留或重新关注了某段内容。

首期事件：

- `page_entered`：进入某页或正文块。
- `meaningful_dwell`：前台可见且超过有效停留阈值。
- `content_revisited`：返回已经读过的位置。
- `highlight_created`：创建高亮。
- `note_created`：创建笔记。
- `question_asked`：用户主动向读伴提问。
- `reading_item_finished`：完成当前阅读项。

约束：

- 滚动抖动、快速掠过和后台窗口停留不计为有效阅读。
- 事件保存在本地，不记录正文全文。
- 事件 schema 需要版本号和数量清理策略。

验收标准：

- 翻页和滚动两种模式都能产生一致的核心事件。
- 窗口失焦后不会继续累计有效停留。
- 旧书和已有阅读进度不会被事件系统改写。

### P7.4 介入调度器

目标：让读伴只在合适且必要的时候开口。

基础判定：

```text
允许介入 = 已到达埋点
        + 满足所需阅读事件
        + 用户未关闭主动提问
        + 不在冷却期
        + 本节未超过上限
        + 该埋点未展示/回答/忽略
```

实施内容：

- 支持 `off`、`moderate`、`active` 三档主动程度。
- 设置跨页冷却、时间冷却、每节上限和同类问题去重。
- 用户选择“稍后”时延迟到下一个合适位置，不立即重复。
- 用户连续忽略后自动降频，但不擅自修改用户设置。
- 阅读项即将结束时，不补发此前错过的全部问题。

验收标准：

- 同一埋点不会重复出现。
- 三档设置产生可观察且可测试的行为差异。
- 快速翻页不会连续触发多个问题。

### P7.5 阅读中提问体验

目标：问题像页边批注一样出现，而不是阻断阅读的系统弹窗。

实施内容：

- 默认在读伴侧栏或页边使用低打扰提示。
- 展开后显示问题、简短提问目的和来源入口。
- 提供回答、稍后和忽略三个明确动作。
- 专注阅读模式下先显示克制提示，不强制展开侧栏。
- 移动端和窄窗口使用覆盖层，但不能遮住用户正在阅读的关键内容。

验收标准：

- 不操作问题也可以继续阅读。
- 用户能查看“为什么现在问”和对应原文。
- 翻页、滚动、专注阅读和窄窗口下均无布局跳动。

### P7.6 回答与记忆闭环

目标：让主动提问产生长期价值，而不是变成一次性聊天。

实施内容：

- 回答关联 `bookId`、`readingItemId`、`anchorId`、来源和时间。
- 读伴可以基于回答进行最多一次必要追问，避免无限问答链。
- 用户可把回答整理为笔记，并保留问题与原文来源。
- 将稳定偏好和重要未解决问题写入本书读伴记忆；不把所有原始回答都塞进 prompt。
- 后续问题可引用用户此前的观点，但必须说明关联来源。

验收标准：

- 回答后可以在本节陪读记录中找到。
- 关闭并重开 App 后关联关系仍然存在。
- 后续上下文使用摘要和索引，不因回答累积无限膨胀。

### P7.7 章节结束沉淀

目标：在完成阅读项时把陪读过程整理成可回看的成果。

实施内容：

- 汇总本节出现的问题、用户回答、忽略项和未解决问题。
- 允许用户选择保留、编辑或删除自动整理内容。
- 生成一条连接下一阅读项的提示，但不提前泄露后续正文细节。
- 在「整理这本书」中提供本节陪读记录入口。

验收标准：

- 用户可以看出本节读伴真正参与了什么。
- 陪读记录能追溯原文，也能继续转为笔记。
- 没有触发任何问题的章节不会生成空洞总结。

### P7.8 开书设置收敛

目标：把“捏读伴”从装饰性定制改为共同确定陪读方式。

首期只保留：

- 阅读目的：理解知识、梳理论证、联系实践、准备考试或自定义。
- 主动程度：关闭、适中、较积极。
- 关注内容：概念、论证、案例、行动与输出。

处理策略：

- 暂停扩展名字、颜色、表情和抽象人格标签。
- 旧 `companionProfile` 数据继续兼容读取，不删除、不强制迁移。
- 每个设置项需要在埋点选择、触发数量或问题形式上有测试可见的差异。
- 设置页明确说明这些选择会怎样影响陪读行为。

验收标准：

- 用户无需理解角色分类即可完成设置。
- 设置结果可以被自动化测试验证，而不只是文案变化。

### P7.9 诊断、指标与 QA

目标：能够判断主动陪读为什么出现、为什么没出现，以及是否过度打扰。

实施内容：

- 开发态查看正文块、埋点、事件、调度判定和最终状态。
- 建立正常文本 PDF、无目录 PDF、扫描页、旧书和长章节固定样本。
- 本地统计候选数、展示数、回答数、稍后数、忽略数和重复拦截数。
- 诊断导出只包含 id、类型、状态、耗时和计数，不包含正文及回答全文。
- 增加 schema、缓存、旧数据兼容和调度器单元测试。

验收标准：

- 测试人员能解释每次介入或未介入的原因。
- 能稳定复现并验证冷却、去重、降频和关闭设置。
- 诊断包通过现有隐私扫描与发布预检。

### P7.10 读伴视觉身份重构

目标：让读伴形象表现已经存在的陪读状态，并与读伴整体气质统一。

实施前提：P7.1-P7.9 已形成稳定行为模型。

实施方向：

- 保留统一的读伴品牌身份，不建设任意五官和服装编辑器。
- 视觉状态映射真实行为：准备问题、等待回答、记录理解、章节沉淀。
- 优先评估纸上小像、藏书票和朱砂印记语言。
- 完整形象只出现在需要建立关系的场景；聊天和页边提示使用简化印记。

验收标准：

- 每种视觉状态都有对应系统状态，不使用无意义表情切换。
- 形象与纸面、墨色、朱砂的 UI 体系一致。
- 动效不影响阅读性能，也不会提高问题打扰感。

## 五、实施顺序与边界

- P7.1 是所有后续步骤的前置条件。
- P7.2 和 P7.3 可以在 P7.1 完成后并行设计，但落库结构必须统一。
- P7.4 依赖真实埋点和事件，不用随机数据直接做生产逻辑。
- P7.5-P7.7 共同构成用户可见闭环，不能只完成问题浮层就宣告主动陪读完成。
- P7.8 可以先做文案和信息架构设计，但应在功能映射明确后再改真实设置流程。
- P7.10 最后进行，避免再次用视觉定制掩盖能力缺口。

首轮只承诺支持文本提取质量合格的 PDF。扫描 PDF、图片/表格理解和 MOBI 细粒度定位分别依赖 OCR、视觉模型或格式解析增强，不纳入 P7-A 的完成门槛。
